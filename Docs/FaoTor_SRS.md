# **SRS: เฝ้าท่อ (FaoTor)**

### **Software Requirements Specification**

**โครงการ:** UTCC AI Hackathon 2026 — Aqua AI & STEM **เวอร์ชันเอกสาร:** 1.0 **วันที่:** 6 กรกฎาคม 2569

---

## **1\. ภาพรวมโครงการ (Project Overview)**

### **1.1 ปัญหา**

ท่อระบายน้ำ/คลองในชุมชนอุดตันด้วยขยะ ทำให้เกิดน้ำท่วมขังซ้ำซากในจุดเดิมทุกปี หน่วยงานท้องถิ่นดูแลแบบ "ตามรอบเวลา" ไม่ใช่ "ตามความเสี่ยงจริง" เพราะไม่มีข้อมูลว่าจุดไหนอุดตันหนักที่สุด

### **1.2 วัตถุประสงค์ของระบบ**

สร้างระบบที่ให้ชุมชนรายงานสภาพท่อผ่านภาพถ่าย แล้วใช้ AI วิเคราะห์ระดับความเสี่ยง เพื่อช่วยเจ้าหน้าที่จัดลำดับความสำคัญในการลอกท่อ ก่อนที่น้ำจะท่วม

### **1.3 กลุ่มผู้ใช้ (User Roles)**

| Role | คำอธิบาย |
| ----- | ----- |
| **ผู้รายงาน (Reporter)** | ชาวบ้าน/ชุมชนที่ถ่ายรูปท่อส่งเข้าระบบ |
| **เจ้าหน้าที่ (Officer/Admin)** | ผู้ดูแลพื้นที่ ดู Dashboard และวางแผนลอกท่อ |

### **1.4 การแบ่ง Phase**

| Phase | เป้าหมาย | Deadline |
| ----- | ----- | ----- |
| **Phase 1 — MVP** | ระบบใช้งานได้จริงขั้นต่ำ สำหรับวิดีโอนำเสนอรอบคัดเลือก | 10 ก.ค. 2569 |
| **Phase 2 — Full Version** | เพิ่มความสมบูรณ์ สำหรับ Prototype รอบชิงชนะเลิศ | 25 ก.ค. 2569 |

## **2\. Requirements — Phase 1 (MVP)**

เป้าหมาย: ทำให้ "Loop หลัก" ของระบบทำงานได้จริง แม้ Data/Design จะยังไม่สมบูรณ์

| ID | Requirement | รายละเอียด |
| ----- | ----- | ----- |
| FR-101 | อัปโหลดรูปท่อ | ผู้ใช้เลือก/ถ่ายรูปท่อ \+ กรอกตำแหน่ง (พิกัดหรือชื่อสถานที่แบบ Manual) |
| FR-102 | วิเคราะห์ภาพด้วย AI | ส่งรูปไปยัง Gemini Vision API (Free Tier) วิเคราะห์ระดับอุดตัน แบ่ง 3 ระดับ: ปกติ / เริ่มอุดตัน / อุดตันหนัก |
| FR-103 | แสดงผลความเสี่ยง | แสดงคะแนน/ระดับความเสี่ยงกลับให้ผู้รายงานเห็นทันทีหลังส่งรูป |
| FR-104 | บันทึกข้อมูล | เก็บรูป \+ พิกัด \+ ระดับความเสี่ยง \+ วันเวลา ลงฐานข้อมูล |
| FR-105 | Dashboard รายการ | หน้าแสดงรายการจุดที่มีรายงาน เรียงจากความเสี่ยงสูง→ต่ำ (ตารางหรือ List ก็พอสำหรับ MVP) |
| FR-106 | อัปเดตสถานะ | เจ้าหน้าที่กดปุ่มเปลี่ยนสถานะ "รอดำเนินการ → กำลังแก้ไข → แก้ไขแล้ว" |

### **Non-Functional (MVP)**

* ต้นทุน: ใช้ Gemini API Free Tier เท่านั้น ห้ามมีค่าใช้จ่าย  
* ใช้งานผ่านเบราว์เซอร์มือถือได้ (Responsive) เพราะชุมชนส่วนใหญ่ใช้มือถือ  
* ไม่ต้องมี Login ระบบสมาชิกที่ซับซ้อน (MVP ใช้ Session/Local ง่ายๆพอ)

## **3\. Requirements — Phase 2 (Full Version)**

เป้าหมาย: เพิ่มความสมบูรณ์และ "เรื่องราว" ให้ Pitch Deck รอบชิงชนะเลิศแข็งแรงขึ้น

| ID | Requirement | รายละเอียด |
| ----- | ----- | ----- |
| FR-201 | แผนที่ความเสี่ยง (Map View) | แสดงจุดเสี่ยงบนแผนที่จริง (เช่น Leaflet/Google Maps) แทนตาราง Text |
| FR-202 | คำนวณ "ความเร่งด่วน" | ผสมคะแนนอุดตัน \+ ข้อมูลพยากรณ์ฝนจาก API สาธารณะ → คะแนนความเร่งด่วนรวม |
| FR-203 | ระบบ Login แยกบทบาท | แยกสิทธิ์ Reporter (ส่งรายงานได้) กับ Officer (จัดการ/ปิดงานได้) |
| FR-204 | ประวัติ/สถิติ | กราฟสรุปจำนวนรายงาน แนวโน้มพื้นที่เสี่ยงตามช่วงเวลา |
| FR-205 | ระบบ Gamification เบื้องต้น | ให้แต้ม/ยศให้ผู้รายงาน เพื่อจูงใจให้ชุมชนรายงานต่อเนื่อง |
| FR-206 | Mockup LINE Bot | ทำหน้า Prototype/Flow แสดงว่าถ้าต่อยอดจริงจะเชื่อมกับ LINE ได้อย่างไร (ไม่ต้อง Deploy จริง) |

### **Non-Functional (Full Version)**

* รองรับผู้ใช้พร้อมกันได้ในระดับ Demo (ไม่ต้องถึงระดับ Production)  
* มี Error Handling พื้นฐานเมื่อ AI วิเคราะห์ภาพไม่ชัดเจน (เช่น "กรุณาถ่ายใหม่ให้เห็นท่อชัดขึ้น")  
* ข้อมูลที่เก็บต้องลบ/แก้ไขได้ในหน้า Admin (Data governance ขั้นต่ำ)

## **4\. System Architecture (สรุปจาก Diagram ที่นำเสนอไปก่อนหน้า)**

| Layer | เทคโนโลยีที่แนะนำ |
| ----- | ----- |
| Frontend | Web App — React หรือ Next.js (Responsive) |
| Backend | Node.js (Express) หรือ Python (FastAPI) |
| AI Vision | Gemini Vision API (Free Tier) |
| ข้อมูลฝน | API สาธารณะจากกรมอุตุนิยมวิทยา/ThaiWater (Phase 2\) |
| Database | Firebase Firestore หรือ PostgreSQL |
| Map (Phase 2\) | Leaflet.js (ฟรี ไม่ต้องผูกบัตร) |

---

## **5\. Data Model (แนวคิดเบื้องต้น)**

**Report (รายงาน)**

* id, image\_url, location (lat/lng หรือชื่อสถานที่), risk\_level (ปกติ/เริ่มอุดตัน/อุดตันหนัก), risk\_score, status (รอดำเนินการ/กำลังแก้ไข/แก้ไขแล้ว), created\_at

**Officer (Phase 2\)**

* id, name, area\_responsible

---

## **6\. Assumptions & Constraints**

* งบประมาณ: 0 บาท ต้องใช้ Free Tier ทุก Service  
* ทีมพัฒนา: Vibe Coding โดยทีมนักเรียน ไม่ใช่ทีม Dev มืออาชีพ → เลือกเทคโนโลยีที่ตั้งค่าง่าย ลด Dependency ที่ซับซ้อน  
* Timeline บีบมาก (Phase 1 เหลือ \~4 วัน) → MVP ต้อง "ทำงานได้จริงบางส่วน" ไม่ใช่ Mockup ปลอมทั้งหมด เพื่อให้วิดีโอนำเสนอน่าเชื่อถือ  
* Phase 2 จะเริ่มได้เฉพาะถ้าผ่านเข้ารอบ 15 ทีม (ประกาศ 15 ก.ค.)

---

## **7\. Success Criteria (เกณฑ์ความสำเร็จของ MVP)**

ระบบ MVP จะถือว่า "พร้อมถ่ายวิดีโอ" เมื่อทำสิ่งต่อไปนี้ได้จริงหน้ากล้อง (ไม่ใช่แค่พูดว่าทำได้):

1. อัปโหลดรูปท่อจริง 1 รูป แล้วเห็น AI ตอบระดับความเสี่ยงกลับมา  
2. เห็นรายการใน Dashboard อัปเดตตามรูปที่ส่งเข้าไป  
3. กดเปลี่ยนสถานะได้ และเห็นผลอัปเดตทันที
